# 工程格式与配置库

## 内置模板与个人工程

工程库同时展示两种来源，但它们具有不同的生命周期：

- **内置模板**随 App 安装，图集和配置从 App Bundle 读取。模板可以显示在桌面、响应动作、复制和导出，但不能改名、编辑动作、归一化或删除；App 更新后会重新使用新版内置内容。
- **个人工程**保存在 `~/Library/Application Support/SpritePetStudio/Projects/<id>/`。每个目录都有独立的 `pet.json`、`studio.json` 和 `spritesheet.png`，可以完整编辑内容与显示名称，也可以复制、导出和删除；内部工程 ID 与目录名保持只读。

首次安装会显示两个内置模板。使用“复制完整工程”后，应用会把模板图集和全部动作配置写入新的个人工程目录；此后所有修改只作用于副本，不会覆盖模板。启动时应用会重新扫描带有 `studio.json` 的个人工程目录，因此模板与用户工程可以独立恢复并同时呈现。

## 工程包

SpritePet Studio 工程使用一张大 PNG，并用 `studio.json` 保存动作布局和播放设置：

```text
my-pet/
├── pet.json
├── spritesheet.png
└── studio.json
```

- `pet.json`：工程 ID、显示名称和图集路径。
- `spritesheet.png`：唯一的帧图集；工程内部没有独立帧目录。
- `studio.json`：工程配置快照、动作播放（包括指定次数）、逐帧草稿调整和触发器（包括可选延迟）。

每个工程还保存自己的桌面显示开关与窗口坐标。它们只属于 SpritePet Studio 运行状态，不影响 PNG 图集，也不妨碍 Codex 忽略 `studio.json` 后继续使用宠物包。

采用内置 Codex v2 配置时，前两个文件与 Codex 宠物包兼容；Codex 会忽略额外的 `studio.json`。自定义配置工程必须保留 `studio.json`。

## 工程 ID 与目录

每个个人工程只维护一套不可编辑的 ID。它同时用于：

- 运行时窗口、触发器和当前工程选择的稳定标识；
- `Projects/<id>/` 工作目录；
- `pet.json` 与 `studio.json` 中的 `id`；
- 导出时生成的顶层文件夹名。

ID 在新建、复制或导入工程时生成一次，之后不能通过设置界面修改。工程的显示名称与描述是独立字段，可以随时编辑，不会影响路径或运行关联。启动扫描个人工作区时，实际目录名是该工程 ID 的权威来源；保存后两个 JSON 会统一写回相同 ID。

## 工程导入与降级

导入面板接受以下任意入口：

- 整个工程文件夹；
- `pet.json` 或旧工程可能使用的 `path.json`；
- `studio.json`；
- `spritesheet.png` 或 `spritesheet.webp`。

应用会从所选文件的同级目录自动发现其余文件。`spritesheet` 是唯一必需内容；两个 JSON 都可以缺失。布局恢复按以下顺序进行：

1. 使用 `studio.json` 中完整且与图片尺寸一致的图集配置；
2. 从 `studio.json` 已有的格位和动作布局推断配置；
3. 根据图片尺寸匹配内置标准：Codex v1 为 1536 × 1872（8 × 9），Codex v2 为 1536 × 2288（8 × 11）；
4. 仍无法识别时，向前端返回缺少的图集文件、损坏的 JSON 或不支持的图片尺寸。

直接选择损坏的 JSON 时，只要同目录仍有可识别图集，也会继续走尺寸降级，不会因为可选元数据失败而阻止导入。PNG 图集会尽量按原始字节复制，其他可读取格式则统一写入个人工程的 `spritesheet.png`。

## 配置库

每个 `AtlasConfiguration` 定义：

- 配置名称、说明和唯一 ID
- 单格宽度与高度
- 有序动作列表
- 每个动作的名称、标签键、总帧数和占用排数

动作按配置顺序连续排入图集。每排格位数自动取所有动作中 `向上取整(总帧数 ÷ 占用排数)` 的最大值；每个动作再按自己的占用排数自动换排。比如 Codex v2 的前 9 个动作各占 1 排，16 方向视线动作占 2 排，因此最终自动得到 8 列 × 11 排。

配置只描述 PNG 图集的物理布局，不包含播放方式、帧率、停留时间、触发器或打断规则。这些运行时行为属于具体工程，在动作编辑器中设置。因此同一份布局配置可以服务于播放逻辑不同的多个工程。

## 触发延迟

每条离散触发规则都可以在 `studio.json` 中保存 `delaySeconds`。它表示事件发生后等待多少秒再开始播放对应动作，默认值为 `0`，因此不含该字段的旧工程会保持原来的即时触发行为。运行时会把负数归零，并把单次延迟限制在 24 小时以内。

桌宠启动、点击、鼠标距离边界、随机、空闲、定时、URL/命令行名称事件和系统事件都支持延迟。鼠标角度跟踪与拖拽方向属于连续实时控制，不应用延迟，以免指向或跑动反馈落后于当前输入。

同一条规则在等待期间再次触发时，会取消旧的等待并按最新一次事件重新计时；停止或重置工程会取消尚未执行的延迟任务。工程库的交互预览与真实桌宠窗口采用相同规则。

## 手动动作预览

动作编辑器中的“播放整个动作”、桌宠窗口右键菜单和 Dock 菜单里的动作列表都属于显式预览入口。无论动作保存的是播放一次、指定次数、持续循环，甚至当前被停用，显式预览都会创建一个不写回工程的临时副本，并：

- 以最高临时优先级播放；
- 从第一帧开始完整播放一遍；
- 在这一遍结束前阻止普通触发器、拖拽跑动和鼠标角度跟踪抢占；
- 允许用户再次选择另一个显式预览立即替换；
- 结束后回到工程默认动作。

该行为只用于检查动作效果，不会改变原动作的启用状态、播放模式、循环次数、优先级或打断规则。

## 工程与配置的关系

每个工程同时保存：

1. 一份完整配置快照，确保脱离当前配置库也能打开；
2. 一个可选的配置库 ID，表示它关联哪一项可复用配置。

导入工程时，如果配置库没有完全相同的配置，应用会询问：

- 加入配置库：保存为可复用配置并关联工程；
- 临时使用：只保留工程内快照，并在工程库显示“临时配置”。

编辑并保存配置库条目时，关联工程会按动作标签键和帧序号迁移已有图像。新增格位保持透明，删除的动作或帧不再进入新图集。

## Codex v1 / v2 默认配置

两套内置标准配置均保持只读，以免意外破坏兼容性：

Codex v1：

- 每排 8 格，共 9 排
- 每格 192 × 208 px
- 整图 1536 × 1872 px
- 9 套标准动作，不包含 16 方向视线动作

Codex v2：

- 每排 8 格
- 11 排
- 每格 192 × 208 px
- 整图 1536 × 2288 px
- 9 套标准动作和 16 个方向帧
- 前 9 套动作各占 1 排，16 方向动作占 2 排

需要修改时先复制对应配置。复制品属于 SpritePet Studio 自定义配置，不再承诺可被 Codex 固定协议识别。

## 单帧编辑与归一化

“导入单帧 PNG”会等比居中写入当前工程配置所指定的格位尺寸，并立即替换大 PNG 中对应格位。

每个 `PetFrameDefinition` 都保存以下非破坏性变换参数：

- `scale`：整体等比缩放，默认 `1`
- `scaleX`：额外横向缩放，默认 `1`
- `scaleY`：额外纵向缩放，默认 `1`
- `offsetX` / `offsetY`：水平与垂直位移，默认 `0`

最终宽度比例为 `scale × scaleX`，最终高度比例为 `scale × scaleY`。旧工程没有 `scaleX` 或 `scaleY` 时会自动按 `1` 读取，保存后写入新字段，因此无需手动转换旧工程。

这些缩放与 X/Y 偏移默认是非破坏性草稿。执行“归一化并写入图集”后，三组缩放和位移会永久烘焙进对应格位，并统一恢复为 `scale = 1`、`scaleX = 1`、`scaleY = 1`、`offsetX = 0`、`offsetY = 0`。停留时间、播放模式和触发器无法编码进 PNG，因此不会被归一化。

“复原参数”与“一键复原”不会修改 `spritesheet.png`，只把上述变换参数恢复为默认值。

动作编辑器支持在同一动作内按住 `Command` 点击多个缩略图。批量编辑不会在工程中创建额外的共享变换：每次 `+1%` 或 `+1 px` 都直接叠加到每个所选帧自己的当前参数，因此不同初始值会保持原有差异。多选时也可以只归一化或复原当前所选帧。

单帧与多帧预览使用固定格位视口，外框、中心十字和像素比例不会随当前缩放或位移自动变化；超出视口的内容会按真实桌宠窗口的效果裁切。叠加常态首帧时也沿用同一个固定坐标系。

工程库中的“一键归一化全部帧”会批量处理该工程所有存在草稿缩放或位移的帧。处理时每个格位都会单独裁切，变换结果不会溢出到相邻格位；未调整的帧保持原始像素。图集发生写入后，桌面宠物、工程预览和单帧预览会在当前运行进程内失效旧缓存并重新载入，无需重启应用。
